# 도시계획분석 서비스 지적 기반 필지 선택 데이터 전략(초안)

- 문서 목적: **한국 서비스 기준**으로, 지적 기반 필지 선택 기능에 필요한 데이터 식별 체계와 저장/조회 전략을 정의한다.
- 적용 범위: MVP~확장 단계의 데이터 전략(연동 구현/DB 구축/분석 로직 구현 제외)
- 핵심 원칙:
  1. 필지 선택의 기준은 **지적(연속지적도) 기반 식별값**으로 통일한다.
  2. 초기 단계에서 전국 원천 데이터를 선적재하지 않는다.
  3. VWorld 원격 조회와 내부 최소 저장을 분리해 운영 복잡도를 낮춘다.
  4. 기존 화면 레이아웃(정보 구조/배치)은 유지하고, 데이터 처리 기능만 단계적으로 업데이트한다.

## 비범위(이번 단계에서 하지 않음)

| 항목 | 제외 이유 | 후속 단계 |
|---|---|---|
| 실제 API 연동 구현 | 전략/계약 정의 단계에 집중 | 원격 조회 어댑터 구현 단계 |
| DB 물리 구축/마이그레이션 | 저장 정책 합의 선행 필요 | 영속 모델 확정 후 |
| GIS 분석 로직 구현 | 필지 선택 데이터 기반 정립이 우선 | 분석 파이프라인 단계 |
| 화면 레이아웃 변경(UI) | 사용자 학습 비용 및 회귀 리스크 최소화 | 별도 UI 개선 과제로 분리 |
| 로그인/권한 기능 구현 | 본 문서 범위 외 | 인증/인가 에픽에서 처리 |

## 1) 필지 선택 기준 데이터

| 구분 | 기준 데이터 | 정의/설명 | 사용 시점 | 필수 여부 | 비고 |
|---|---|---|---|---|---|
| 1차 식별 | PNU(19자리) | 법정동코드+산여부+본번+부번을 결합한 표준 필지 식별자 | 지도 클릭 후 필지 확정, API/내부 키 매핑 | 필수 | 서비스 내부의 canonical key 권장 |
| 2차 식별 | 법정동코드(10자리) | 행정구역 기반 법정동 단위 코드 | 검색/필터/통계 집계 | 필수 | 동일 본번/부번 충돌 방지에 필요 |
| 3차 식별 | 본번/부번 | 지번 구성 번호 | 사용자 표시, 검색조건 | 필수 | 숫자형 저장 + 표시 시 문자열 포맷 |
| 보조 식별 | 산여부(대지/산) | 산지 지번 여부 구분 | 지번 문자열 생성, PNU 생성/검증 | 필수 | UI 표시는 “산” 접두 처리 |
| 공간 참조 | 필지 경계 Geometry | 선택 필지 폴리곤/멀티폴리곤 | 지도 하이라이트, 면적 재계산 | 선택(MVP는 원격 우선) | 대용량이므로 선별 캐시 권장 |
| 위치 참조 | 중심점(Point) | 필지 대표 좌표 | 목록 핀 표시, 재조회 | 선택 | 경계 미저장 시 경량 대체값 |

### 운영 기준

| 항목 | 권장안 |
|---|---|
| 필지 선택 확정 키 | PNU 단일 키 사용 |
| 사용자 표시 기본값 | `법정동명 + 산여부 + 본번-부번` |
| 중복 처리 | 동일 PNU 중복 선택 방지 |
| 좌표계 처리 | 수신 좌표계 명시 저장(예: EPSG 코드), 내부 표준 좌표계 1개로 정규화 |

## 2) 필지 식별 체계 정리 (PNU/법정동코드/본번/부번)

## 2-1. 필드 정의 표

| 필드명 | 길이/형식 | 예시 | 설명 | 검증 규칙 |
|---|---|---|---|---|
| pnu | 19자리 문자열 | 1168010100101230000 | 필지 유일 식별자(권장 내부 기본키) | 숫자 19자리, 구성요소 파싱 가능해야 함 |
| bjd_code | 10자리 문자열 | 1168010100 | 법정동코드 | 숫자 10자리 |
| mountain_yn | 1자리(0/1) | 0 | 산여부(0: 대지, 1: 산) | 0/1만 허용 |
| main_no | 4자리(표준) | 0123 | 본번 | 0 이상 정수, 표시 시 zero-pad 정책 통일 |
| sub_no | 4자리(표준) | 0000 | 부번(없으면 0000) | 0 이상 정수 |
| lot_label | 가변 문자열 | 역삼동 123-0 | 사용자 표시용 지번 문자열 | 입력값 직접 저장보다 생성 규칙 기반 권장 |

## 2-2. 조합/표시 규칙 표

| 규칙 구분 | 규칙 |
|---|---|
| PNU 생성 | `bjd_code(10) + mountain_yn(1) + main_no(4) + sub_no(4)` |
| 표시 지번 생성 | `산` 접두(산여부=1일 때) + 본번 + (부번>0 이면 `-부번`) |
| 입력 정규화 | 숫자 외 문자 제거 후 길이/범위 검증 |
| 저장 정규화 | `pnu`는 문자열 고정, 본번/부번은 정수 + 파생 문자열 분리 |
| 교차 검증 | pnu 파싱값과 별도 필드(bjd_code/main/sub/mountain) 일치 여부 검증 |

## 2-3. 실무 권장 데이터 타입 표

| 항목 | API 요청/응답 | 내부 저장(권장) | 이유 |
|---|---|---|---|
| pnu | string | string(19) | 선행 0 보존 및 외부 시스템 호환 |
| bjd_code | string | string(10) | 코드성 데이터 정합성 |
| main_no | number/string | integer + display string(optional) | 정렬/범위검색과 표시 품질 동시 확보 |
| sub_no | number/string | integer + display string(optional) | 동일 |
| geometry | GeoJSON/WKT 등 | MVP 비영속 또는 단기 캐시 | 용량/갱신비용 절감 |

## 3) VWorld 원격 조회 데이터 vs 내부 저장 데이터

## 3-1. 책임 분리 표

| 구분 | VWorld 원격 조회(실시간/준실시간) | 우리 서비스 내부 저장(최소) |
|---|---|---|
| 필지 경계 | 연속지적도 기반 경계 geometry 조회 | 기본은 미저장, 필요 시 세션/단기 캐시 |
| 필지 속성 | 지번 관련 속성(PNU, 본번/부번, 산여부, 법정동 코드/명) 조회 | 사용자 선택 결과의 정규화된 식별값만 저장 |
| 검색 | 주소/지번 검색 API 활용 | 검색어 로그/선택 이력(비식별/최소화)만 저장 |
| 최신성 관리 | 원격 데이터 최신 상태를 조회 시점에 반영 | 내부는 비즈니스 상태(즐겨찾기/프로젝트 연결 등) 위주 |
| 장애 대응 | 외부 API 오류/지연 가능 | 마지막 성공 선택값, 최소 fallback 메시지 저장 |

## 3-1-1. 조회/저장 결정 기준 표

| 판단 기준 | 원격 조회 우선 조건 | 내부 저장 우선 조건 |
|---|---|---|
| 최신성 민감도 | 최신 지적 상태 반영이 중요할 때 | 사용자 확정 결과를 재현해야 할 때 |
| 데이터 크기 | Geometry 등 대용량 원본일 때 | 소형 식별자/메타데이터일 때 |
| 재사용 빈도 | 단발성 조회가 대부분일 때 | 동일 프로젝트에서 반복 참조될 때 |
| 장애 내성 | 일시 장애 시 재시도 가능할 때 | 장애 시 즉시 보여줄 최소 상태가 필요할 때 |
| 법/운영 리스크 | 원본 장기보관 부담이 클 때 | 감사/추적을 위한 최소 로그가 필요할 때 |

## 3-2. 내부 최소 저장 항목 표 (MVP 기준)

| 분류 | 저장 항목 | 필요성 | 보관 범위 |
|---|---|---|---|
| 필수 식별 | pnu, bjd_code, mountain_yn, main_no, sub_no | 재조회/재현 가능성 확보 | 영속 저장 |
| 사용자 맥락 | 프로젝트ID-필지 매핑, 선택 시각 | 사용자 작업 연속성 | 영속 저장 |
| 표시 최적화 | lot_label(파생), center_point(optional) | 목록/지도 UX 성능 향상 | 선택 저장 |
| 운영 로그 | 외부 조회 성공/실패 코드, 응답시간(집계) | 장애 분석/튜닝 | 집계성 저장 |
| 대용량 공간 | 원본 geometry | MVP에서는 비용 대비 효익 낮음 | 기본 미저장 |

## 3-3. 저장 제외(기본) 항목 표

| 항목 | MVP 기본 방침 | 사유 |
|---|---|---|
| 전국 필지 원천 전수 적재 | 제외 | 초기 구축비용/운영비 과대 |
| 원격 원본 전문(raw payload) 장기보관 | 제외 | 스키마 변동/저장비용/보안 이슈 |
| 전체 필지 경계 장기 캐시 | 제외 | 갱신 정책 복잡도 높음 |

## 4) 단계별 데이터 전략 (MVP vs 확장)

## 4-1. 단계 전략 요약 표

| 단계 | 목표 | 데이터 접근 | 내부 저장 정책 | 기대 효과 | 리스크/대응 |
|---|---|---|---|---|---|
| MVP | 필지 선택 정확도 확보 + 빠른 출시 | VWorld 검색/연속지적도 원격 조회 중심 | 필지 식별자(PNU 계열) + 사용자 선택 맥락 최소 저장 | 개발 속도/운영 단순성 | 외부 의존도↑ → 재시도/타임아웃/사용자 안내 |
| 확장 1 | 성능/안정성 개선 | 조회 빈도 높은 지역/필지 선별 캐시 | 중심점/경계 일부 캐시, TTL 운영 | 응답속도 개선 | 캐시 불일치 → 만료/검증 배치 |
| 확장 2 | 분석 활용도 확대 | 주제도/규제 레이어 연계 | 분석 결과물/버전 메타데이터 저장 | 분석 재현성 강화 | 데이터 계보 관리 필요 |
| 확장 3 | 운영 고도화 | 다중 소스 융합(공공/민간) | 표준 스키마/품질지표 저장 | 장애 내성/확장성 | 표준화 비용 증가 |

## 4-2. MVP 상세 전략 체크리스트 표

| 항목 | MVP 결정 | 구현 시 메모 |
|---|---|---|
| 필지 식별 표준 | PNU canonical | 모든 내부 이벤트에 PNU 포함 |
| 검색/선택 흐름 | 원격 조회 후 사용자 확정 | 확정 시점 데이터만 영속화 |
| Geometry 저장 | 기본 미저장 | 필요 시 세션 캐시(짧은 TTL) |
| 데이터 정합성 | pnu↔세부필드 교차검증 | 불일치 시 저장 차단/로그 |
| 관측성 | API 성공률/지연 계측 | 운영 대시보드 지표화 |

## 4-3. UI 레이아웃 유지하며 기능 업데이트 원칙

| 구분 | 원칙 | 적용 예시 |
|---|---|---|
| 화면 구조 | 기존 패널/지도/리스트 배치는 유지 | 지도 클릭→우측 정보패널 갱신만 변경 |
| 사용자 동선 | 기존 클릭 수/확정 흐름 유지 | “필지 선택 확정” 버튼 위치·용어 고정 |
| 데이터 바인딩 | 표시 필드만 교체/추가, 컴포넌트 골격 유지 | 내부 키를 PNU로 전환하되 카드 레이아웃 유지 |
| 호환성 | 기존 화면 이벤트 계약을 우선 보존 | `onParcelSelect` 페이로드에 pnu 추가, 기존 필드 유지 |
| 회귀 방지 | 기능 변경 시 레이아웃 스냅샷/체크리스트로 검증 | CSS/배치 수정 금지 항목 명시 |

## 5) 다음 구현 단계 제안

| 우선순위 | 작업 항목 | 산출물 | 비고 |
|---|---|---|---|
| 1 | 필지 식별 DTO/스키마 계약서 작성 | API 필드 명세서(PNU/법정동/본번/부번) | 프론트/백엔드 공통 계약 |
| 2 | 원격 조회 어댑터 인터페이스 정의 | provider interface + 오류코드 표준 | 실제 API 연동 구현은 다음 단계 |
| 3 | 선택 결과 영속 모델 정의 | 최소 저장 테이블 설계서(논리) | 물리 DB 구축은 다음 단계 |
| 4 | 데이터 검증 규칙 정의 | 정규화/검증 유닛 테스트 케이스 목록 | 구현 전 테스트 기준점 확보 |
| 5 | 캐시 도입 기준 정의 | 캐시 대상/TTL/무효화 정책 문서 | 확장 1 단계 진입 조건 |

## 부록 A. 의사결정 요약

| 질문 | 결정 |
|---|---|
| 필지 선택 기준 데이터는? | PNU 중심 + 법정동코드/본번/부번/산여부 보조 |
| 외부/내부 데이터 경계는? | 경계/검색은 VWorld 원격 우선, 내부는 선택 결과 최소 저장 |
| 전국 데이터 선적재는? | 기본안에서 제외 |
| MVP와 확장의 차이는? | MVP는 정확한 식별과 경량 저장, 확장은 성능/분석/내재화 순차 강화 |
