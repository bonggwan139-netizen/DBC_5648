# 도시계획분석 웹사이트 지적 기반 필지 선택 데이터 전략(초안)

- 작성 목적: **한국 서비스 기준**으로, 지적 기반 필지 선택 기능의 기준 데이터/식별 체계/원격-내부 데이터 경계를 실무적으로 정리한다.
- 적용 범위: MVP 및 확장 단계 데이터 전략 문서화(※ API 연동 구현, DB 구축, GIS 분석 로직 구현, 화면/로그인 수정은 범위 제외)
- 설계 원칙:
  1. 필지 선택의 canonical key는 **PNU(19자리)**로 통일한다.
  2. 초기 기본안은 **전국 데이터 선적재를 채택하지 않는다**.
  3. VWorld(연속지적도/검색)는 원격 조회 중심으로 활용하고, 내부에는 서비스 운영에 필요한 최소 데이터만 저장한다.

## 1) 필지 선택 기준 데이터 정의

### 1-1. 필지 선택 기준 데이터 표

| 구분 | 데이터 | 설명 | 선택 확정 시 필요 | 저장 기본 정책(MVP) | 비고 |
|---|---|---|---|---|---|
| 기준 식별자 | PNU(19) | 법정동코드+산여부+본번+부번 결합 식별자 | 필수 | 영속 저장 | 내부 canonical key |
| 행정 식별 | 법정동코드(10) | 법정동 단위 코드 | 필수 | 영속 저장 | 필터/통계/검증에 필요 |
| 지번 구성 | 본번/부번 | 지번 번호 체계 | 필수 | 영속 저장 | UI 표기 및 재현에 필요 |
| 보조 구분 | 산여부(0/1) | 일반지/산지 구분 | 필수 | 영속 저장 | 지번 표기 규칙 영향 |
| 공간 데이터 | 필지 경계 Geometry | 폴리곤 경계 | 권장(표시용) | 기본 미저장(단기 캐시) | 용량·갱신 부담 큼 |
| 경량 좌표 | 중심점(Point) | 목록/지도 이동용 대표점 | 선택 | 선택 저장 | 성능 최적화 용도 |
| 표시 텍스트 | 지번 라벨 | 예: 역삼동 산123-4 | 권장 | 파생 저장 가능 | 원천보다 생성 규칙 권장 |

### 1-2. 선택 이벤트 최소 페이로드(권장)

| 항목 | 타입 | 예시 | 목적 |
|---|---|---|---|
| pnu | string(19) | `1168010100101230004` | 필지 유일 식별 |
| bjd_code | string(10) | `1168010100` | 코드 검증/집계 |
| mountain_yn | string(1) | `0` | 지번 생성 규칙 |
| main_no | int | `123` | 지번 구성 |
| sub_no | int | `4` | 지번 구성 |
| source_provider | string | `vworld` | 원천 추적 |
| selected_at | datetime | `2026-04-14T09:10:00Z` | 이력/재현 |

## 2) 필지 식별 체계 정리 (PNU/법정동코드/본번/부번)

### 2-1. 식별 체계 상세 표

| 필드 | 자리수/형식 | 도메인 규칙 | 검증 포인트 | 내부 저장 권장 타입 |
|---|---|---|---|---|
| pnu | 19자리 숫자 문자열 | `[법정동코드10]+[산여부1]+[본번4]+[부번4]` | 길이/숫자 여부/분해 일치 | `CHAR(19)` 또는 string |
| bjd_code | 10자리 숫자 문자열 | 법정동 코드 체계 준수 | 길이/숫자 여부 | `CHAR(10)` 또는 string |
| mountain_yn | 1자리(`0/1`) | `0`: 일반, `1`: 산 | enum 범위 | `CHAR(1)` 또는 tinyint |
| main_no | 1~4자리 숫자 | 본번은 0 이상 | 음수 불가/범위 검증 | integer |
| sub_no | 1~4자리 숫자 | 부번 미존재 시 0 | 음수 불가/범위 검증 | integer |

### 2-2. PNU 분해/조합 규칙 표

| 규칙 | 식 | 실무 메모 |
|---|---|---|
| PNU 조합 | `pnu = bjd_code + mountain_yn + main_no(4pad) + sub_no(4pad)` | pad 정책을 서버 단일화 |
| 본번 표시 | `main_no` | `0123` 저장 대신 숫자 저장 + 표기 시 포맷 |
| 부번 표시 | `sub_no==0` 이면 생략, 아니면 `-sub_no` | UI 일관성 중요 |
| 산 표기 | `mountain_yn==1` 이면 `산` 접두 | 사용자 인지성 향상 |
| 교차 검증 | 입력 pnu와 분해값 비교 | 불일치 시 저장 차단 |

### 2-3. 사용자 표시 문자열 생성 규칙 표

| 입력 조건 | 출력 예시 |
|---|---|
| 일반지, 부번 없음 (`mountain_yn=0, main=123, sub=0`) | `123` |
| 일반지, 부번 있음 (`0, 123, 4`) | `123-4` |
| 산지, 부번 없음 (`1, 55, 0`) | `산55` |
| 산지, 부번 있음 (`1, 55, 2`) | `산55-2` |

## 3) VWorld 원격 조회 vs 내부 저장 데이터 경계

### 3-1. 책임 분리 표 (Remote vs Internal)

| 영역 | VWorld 원격 조회(권장) | 내부 저장(권장) | 이유 |
|---|---|---|---|
| 필지 검색 | 지번/주소 기반 검색 결과 | 검색어 원문(선택), 선택된 결과 식별자 | 검색 인덱스 내재화 비용 절감 |
| 필지 경계 | 연속지적도 경계 geometry 조회 | 기본 미저장(필요 시 TTL 캐시) | 대용량/최신성 부담 최소화 |
| 필지 속성 | PNU, 법정동명/코드, 본번/부번, 산여부 | 선택 확정 필드(정규화)만 저장 | 재현 가능 + 저장 최소화 |
| 좌표/지도 | 원격 좌표/geometry | center_point 선택 저장 | 목록 이동 성능 보조 |
| 오류/품질 | 원격 응답 코드/지연 시간 | 집계 메트릭(성공률, 지연 p95) | 운영 안정화 지표 확보 |

### 3-2. 내부 저장 데이터 최소안(MVP)

| 데이터 분류 | 저장 항목 | 필수 여부 | 보관 정책 |
|---|---|---|---|
| 필지 식별 | pnu, bjd_code, mountain_yn, main_no, sub_no | 필수 | 영속 저장 |
| 업무 매핑 | project_id, user_id(또는 주체), selected_at | 필수 | 영속 저장 |
| 표시 최적화 | lot_label, center_point | 선택 | 필요 시 저장 |
| 운영 관측 | provider, status_code, latency_ms(집계) | 권장 | 집계 중심 저장 |
| 원격 원문 | raw payload 전체 | 비권장 | 기본 미저장 |

### 3-3. 저장 제외 기본안 표

| 제외 항목 | 기본 방침 | 판단 근거 |
|---|---|---|
| 전국 필지 원천 전수 적재 | 제외 | 초기 비용/복잡도 과다 |
| 전국 경계 대량 캐시 | 제외 | 갱신 및 정합성 관리 어려움 |
| 원격 응답 전문 장기보관 | 제외 | 스키마 변동/보안/비용 이슈 |

## 4) 단계별 데이터 전략 (MVP vs 확장)

### 4-1. 단계 구분 표

| 단계 | 목표 | 데이터 접근 전략 | 내부 저장 전략 | 진입/전환 기준 |
|---|---|---|---|---|
| MVP | 정확한 필지 선택 + 빠른 배포 | VWorld 원격 조회 중심(검색/경계) | 식별자+업무매핑 최소 저장 | 사용자 검증 완료, 기본 성능 충족 |
| 확장 1(성능) | 응답 안정화 | 고빈도 지역/필지 선별 캐시 | center/geometry 부분 캐시(TTL) | API 지연/비용 임계치 초과 |
| 확장 2(분석) | 분석 재현성 강화 | 선택 필지 기반 주제도 결합 | 분석결과/버전 메타 저장 | 분석 기능 상용화 필요 |
| 확장 3(내재화) | 외부 의존도 완화 | 다중 소스 병행/보조 소스 도입 | 표준 스키마/품질지표 고도화 | SLA/비용 최적화 목표 |

### 4-2. MVP 전략 요약 표 (실행 체크)

| 체크 항목 | MVP 결론 | 운영 체크 포인트 |
|---|---|---|
| canonical key | PNU 단일화 | 전 API 이벤트에 pnu 포함 |
| 데이터 최소 저장 | 식별+업무매핑 중심 | geometry 미저장 원칙 유지 |
| 정합성 검증 | pnu 분해값 교차검증 | 불일치율 모니터링 |
| 장애 대응 | 재시도/타임아웃/안내 문구 | provider 장애율 모니터링 |
| 확장 여지 | 캐시/분석 저장은 후속 단계 | 임계치 기반 단계 전환 |

## 5) 다음 구현 단계 제안 (문서 이후)

| 우선순위 | 작업 | 산출물 | 이번 범위와의 관계 |
|---|---|---|---|
| 1 | 필지 식별 스키마 계약 확정 | API 필드 명세서(PNU/bjd/main/sub/mountain) | 구현 전 필수 준비 |
| 2 | VWorld 조회 어댑터 인터페이스 설계 | provider interface + 오류코드 표준표 | 연동 구현의 선행 설계 |
| 3 | 선택 결과 저장 논리 모델 설계 | 논리 ERD/컬럼 정의서(최소 저장 중심) | 물리 DB 구축 전 단계 |
| 4 | 검증/정규화 테스트 케이스 정의 | PNU 파싱/교차검증 케이스 목록 | 품질 기준선 확보 |
| 5 | 캐시 도입 기준 문서화 | TTL/무효화/대상 선정 기준 | 확장 1 진입 기준 수립 |
