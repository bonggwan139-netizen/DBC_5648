# 전체 아키텍처 개요 (현재 단계)

## 문서 목적
이 문서는 현재 프로젝트의 전체 구조를 **짧고 명확하게** 공유하기 위한 아키텍처 개요 문서입니다.

## 현재 아키텍처 한눈에 보기
- **프론트엔드(Web)**: `apps/web` 중심의 Next.js 기반 UI 실험 레이어
- **백엔드(API)**: `apps/api` 중심의 FastAPI 골격 레이어
- **GIS**: 아직 구현 전이며, 추후 `features/map` 또는 API GIS 모듈로 확장 예정
- **데이터(DB/저장소)**: 현재 미연결(스키마/마이그레이션 미정)
- **배포(Infra/운영)**: 현재 실험 우선 단계로, 운영 배포 설계/자동화는 후순위

## 영역별 구조 요약
### 1) 프론트엔드
- 위치: `apps/web`
- 역할: 서비스 화면, 라우팅, 공통 UI 렌더링
- 상태: 페이지 골격 중심, 핵심 UX 흐름 빠른 검증 목적

### 2) 백엔드
- 위치: `apps/api`
- 역할: 향후 도메인 API, 비즈니스 로직, 외부 연동의 진입점
- 상태: 모듈형 확장을 고려한 초기 골격만 준비

### 3) GIS
- 위치(예정): `apps/web/src/features/map`, `apps/api` 내 GIS 관련 모듈
- 역할: 지도 시각화, 공간분석 API 연계
- 상태: **현재 단계 미구현**

### 4) 데이터
- 위치(예정): API 내부 repository/service 계층 + 외부 DB
- 역할: 분석 데이터 적재/조회, 결과 저장
- 상태: **현재 단계 미연결**

### 5) 배포
- 위치(예정): `infra` 및 CI/CD 파이프라인
- 역할: 빌드/배포 자동화, 운영 환경 구성
- 상태: **현재 단계 미구현(로컬 실험 우선)**

## 현재 제품/개발 원칙
- **로그인 기능은 현재 미구현**으로 유지한다.
- **무과금(저비용) 실험 우선** 원칙으로, 로컬 실행과 최소 의존성 기반으로 검증한다.
- 기술 확정 전에는 과도한 세부 설계(배포 고도화, DB 상세 모델링)를 진행하지 않는다.

## 향후 확장 포인트 (아주 짧게)
- 인증/권한: `auth` 모듈 추가
- GIS 기능: 지도 렌더링 + 공간분석 API 분리
- 데이터 계층: 저장소 추상화 및 DB 연결
- 배포: 환경별 설정(dev/stg/prod)과 CI/CD 연결

## 핵심 결정사항 요약 표
| 결정 항목 | 현재 결정 | 이유 |
|---|---|---|
| 로그인 | 미구현 유지 | 초기 UX/기능 검증 속도 우선 |
| 개발 전략 | 무과금 실험 우선 | 비용 부담 없이 빠른 반복 |
| 프론트/백엔드 구조 | `apps/web` + `apps/api` 분리 | 역할 분리 및 확장성 확보 |
| GIS | 추후 모듈화 도입 | 라이브러리/요구사항 확정 전 유연성 확보 |
| 데이터 | 현재 미연결 | 상세 스키마 설계보다 기능 검증 선행 |
| 배포 | 운영 배포 후순위 | 지금 단계 목표가 실험/검증이기 때문 |
